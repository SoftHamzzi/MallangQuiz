## Q1. 🟡 보통

**문제:** 루프에서 변수를 정의할 때 **루프 바깥에서 정의하는 A 방법**과 **루프 안에서 정의하는 B 방법**의 생성자·소멸자·대입 연산자 호출 횟수를 각각 비교하고, 어떤 상황에서 어떤 방법이 유리한지 설명하라.

**정답:**

| 방법 | 생성자 | 소멸자 | 대입 |
|---|---|---|---|
| A (루프 바깥) | 1번 | 1번 | n번 |
| B (루프 안) | n번 | n번 | 없음 |

- **A 방법이 유리한 경우:** 대입 연산 비용이 생성자+소멸자 비용보다 작고, 성능에 민감한 코드일 때
- **B 방법이 유리한 경우:** 대입 연산 비용이 생성자+소멸자 비용보다 크거나 같을 때, 또는 변수의 유효 범위를 루프 내부로 좁혀 유지보수성을 높이고 싶을 때

기본적으로는 **B 방법**을 선호하는 것이 좋다. 유효 범위가 좁아 코드가 명확해지고, 불필요한 상태 노출을 막는다.

**비동기 입출력 처리 방식 4가지:**
1. 논블로킹 소켓 방식
2. Overlapped I/O 방식
3. epoll 방식
4. I/O Completion Port (IOCP) 방식

## Q3. 🔴 어려움

**문제:** C++에서 `is-a`, `has-a`, `is-implemented-in-terms-of` 세 가지 관계를 각각 어떤 구현 방법으로 표현하는지 설명하라. 또한 `Set : public list<T>` 코드가 왜 논리적으로 잘못된 설계인지 근거를 들어 설명하라.

**정답:**

| 관계 | 구현 방법 | 사용 영역 |
|---|---|---|
| is-a (…는 …의 일종) | public 상속 | - |
| has-a (…는 …를 가짐) | 객체 합성(composition) | 응용 영역 |
| is-implemented-in-terms-of | 객체 합성(composition) | 구현 영역 |

**`Set : public list<T>`가 잘못된 이유:**
- public 상속은 `is-a` 관계를 주장한다 → "Set은 list의 일종이다"
- 그러나 list는 **중복 원소를 허용**하지만, Set은 **중복 원소를 허용하지 않는다**
- 논리적으로 모순이 되므로 is-a 관계가 성립하지 않는다
- 올바른 구현은 `list<T>`를 private 멤버로 갖는 **객체 합성**(`is-implemented-in-terms-of`)을 사용해야 한다

## Q4. 🟢 쉬움

**문제:** 게임 서버 개발에서 **RDBMS와 NoSQL을 혼용**하는 이유를 두 DB의 특성 차이와 함께 설명하고, 각각이 주로 어떤 목적으로 사용되는지 서술하라.

**정답:**

| 구분 | RDBMS | NoSQL |
|---|---|---|
| 레코드 구조 | 테이블 내 모든 레코드가 동일한 구조 | 컬렉션 내 도큐먼트마다 다른 구조 가능 |
| 중요점 | 데이터 원자성·일관성 | 가용성·수평 확장성 |
| 질의 기능 | 풍부하고 표준화됨(SQL) | 상대적으로 적고 표준 없음 |

**혼용 이유 및 주 사용 용도:**
- **RDBMS:** 게임 플레이어 데이터(캐릭터 정보, 아이템, 재화 등) 관리 — 일관성과 정확성이 중요
- **NoSQL:** 로그 분석, 통계 분석 — 대량 데이터의 빠른 쓰기/읽기와 유연한 구조가 필요
- 두 DB 각각의 강점이 다르기 때문에 단일 엔진으로는 모든 요구사항을 충족하기 어려워 혼용한다

## Q5. 🟡 보통

**문제:** 스레드 풀링(Thread Pooling)이 필요한 이유를 설명하고, CPU 연산 위주의 작업과 IO 대기가 많은 작업(75% 대기)에서 적정 스레드 개수를 어떻게 결정해야 하는지 설명하라.

**정답:**

**스레드 풀링이 필요한 이유:**
- 클라이언트마다 스레드를 생성하면 수천 개의 스레드가 만들어질 수 있다
- 스레드가 많아지면 메모리 낭비 + 컨텍스트 스위치 비용이 폭증하여 실제 작업보다 전환 비용이 더 커진다
- 스레드 풀은 미리 고정된 수의 스레드를 만들어두고 작업을 분배함으로써 이 문제를 해결한다

**스레드 개수 결정:**

| 작업 특성 | 권장 스레드 수 |
|---|---|
| CPU 연산 위주 | CPU 코어 수와 동일하게 |
| IO 대기 75% (CPU 25%) | CPU 코어 수 × 4 (3~4배) |

- IO 대기 중 CPU는 놀고 있으므로, 스레드 수를 늘려 다른 작업이 CPU를 활용할 수 있도록 한다
- CPU 코어 수보다 스레드가 지나치게 많으면 컨텍스트 스위치가 늘어나 오히려 성능이 저하된다

> **출처:** `server/game_server/1/2025-06-16-game_server_1_12.md`
